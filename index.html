Почему это важно сейчас?

Текущий подход к построению интерфейса сложился исторически. С ростом проекта системные проблемы усугубляются - и напрямую тормозят разработку, снижают качество продукта, увеличивают стоимость изменений

📉
Снижает качество

💸
Дорожает с каждым спринтом
Дизайн-долг копится быстрее, чем отдаётся
Боль #1

Архитектура «колодцев» - всё во viewport

Виджеты проектируются в расчёте на видимую область → каждый обзаводится своим внутренним скроллом. Интерфейс превращается в набор независимых колодцев

Viewport [↕ scroll] ага, и тут тоже
└─ Виджет A [↕ scroll]
└─ Виджет B [↕ scroll]
└─ Виджет C [↕ scroll]
✗ UX кошмар на маленьких экранах
✗ Нарушение доступности
✗ Поведение при resize не предусмотрено
✗ Адаптив «на глаз» - расхождение макет/прод
Боли #2 и #3

Адаптив без правил & разрыв Figma → Прод

📐
Резиновый дизайн без методологии
Fluid-подход имеет свои ограничения. Без брейкпоинтов и правил - ломается. Адаптив де-факто делается фронтендом «по ситуации»:
→ Инконсистентность одних виджетов на разных страницах
→ Споры в ревью «как это на 1280?»
→ Компоненты сложно переиспользовать
🔀
Разрыв бизнес → дизайн → разработка
Дизайнер приступает без полного понимания требований → цикл макет → замечания → переработка:
→ Прод и Figma живут собственными жизнями
→ Фронтенд принимает дизайн-решения
→ Дизайн-долг + технический долг одновременно

01 - UI-кит

Единственный источник истины

🎨
Дизайн токены
Типографика, цвета, отступы, тени, радиусы
📏
Брейкпоинты
Явно зафиксированные, согласованные с командой фронтенда
🧩
Примитивы
Кнопки, инпуты, бейджи, иконки - со всеми состояниями: default, hover, focus, disabled, error
🚫
Правило скролла
Вложенный скролл = только обоснованное исключение. Нативный скролл страницы - дефолт
⬛
Сетка и лейаут
Column grid, gutter, margin - для каждого брейкпоинта
🔗
Figma Tokens Plugin
Токены синхронизируются между Figma и Chakra UI автоматически

Виджет растёт с контентом - не обрезает его. Перекомпоновка прописана в макете явно.

💻

Desktop
🖥️
Wide
Wide Desktop

✓ Приоритет - нативный скролл страницы
✓ Документация в Notion / Confluence / Storybook
03 - Паспорт компонента

Чеклист перед разработкой

🟡 Бизнес
✓
Бизнес-требование сформулировано
✓
User Story понятна и зафиксирована
🟣 Дизайн
✓
Edge cases: пусто, ошибка, длинный текст
✓
Все интерактивные состояния описаны
✓
Адаптив на всех брейкпоинтах в макете
✓
Нет необоснованных вложенных скроллов
✓
Соответствует согласованному UI-киту
🩵 Разработка
✓
FSD-слой определён (widget/feature/entity)
✓
Нет магических значений вне токенов
✓
Resize без кастомной логики вне UI-кита

Паспорт встроен в PR template и Definition of Done - иначе он просто «лежит в Notion»

04 - Роадмап

Поэтапно, без остановки разработки

Этап 0 Аудит 1–2 нед
Фиксируем расхождения фронта с макетом и все вложенные скроллы
Этап 1 Фундамент 2–4 нед
Синхронизация токенов Figma ↔ Chakra UI. Фиксация брейкпоинтов. Визуал не меняется
Этап 2 UI-кит + Паспорт параллельно
Новые компоненты - только по паспорту. Старые обновляются по мере попадания в спринт
Этап 3–4 Адаптив + Миграция инкрементально
Описание поведения типовых виджетов → инкрементальная миграция приоритетных проблемных компонентов
Результат

Что получаем

🌱
Не ломаем - эволюционируем
Продукт продолжает развиваться без «дизайн-стопа»
🔮
Предсказуемость для фронта
Виджет имеет описанные паттерны поведения - не надо придумывать адаптив «на ходу»
💬
Единый язык команды
Токены, брейкпоинты и паспорт - общий словарь для дизайна, фронта и бизнеса
📦
Снижение дизайн-долга
Паспорт отсекает недодуманные макеты ещё до старта разработки
♿
A11y как побочный эффект
Отказ от вложенных скроллов + явные состояния = хорошая база для WCAG
🔄
Инкрементальное внедрение
Новые модули сразу по правилам; старые подтягиваются параллельно фич-разработке
Риски

Честно о подводных камнях

⚠️
Паспорт воспринимается как бюрократия → Чеклист: 10–15 пунктов, не 40. Объяснить ценность команде
⚠️
Токены снова разойдутся → Нужен процесс синхронизации. Либо сами, либо автоматизация через Style Dictionary
⚠️
Чеклист надо согласовать → Встроить в бизнес требования и Definition of Done - не оставлять как «рекомендацию»
⚠️
Инконсистентность в переходный период → Честно принять и зафиксировать срок переходного периода
⚠️
Время дизайнера → Приоритет только на типовых виджетах или выделить время явно
✨

Методология - это не ограничение.
Это то, что даёт команде скорость.
